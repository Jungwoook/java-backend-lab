# 도메인 모델링 기초

## 왜 배우는가

객체지향 설계는 현실의 개념을 그대로 코드로 옮기는 일이 아니다. 중요한 것은 문제 영역에서 의미 있는 개념을 찾아 객체로 표현하고, 각 객체가 책임을 가지도록 만드는 것이다.

도메인 모델링은 회원, 주문, 상품, 결제처럼 프로그램이 다루는 문제 영역을 코드 구조로 옮기는 과정이다.

Spring MVC와 JPA를 배우면 Entity, Service, Repository 같은 구조를 자주 만들게 된다. 이때 도메인 모델링을 이해하지 못하면 모든 비즈니스 로직이 Service에 몰리기 쉽다.

## 개념

도메인은 소프트웨어가 해결하려는 문제 영역이다.

도메인 모델은 그 문제 영역의 중요한 개념과 규칙을 코드로 표현한 것이다.

예를 들어 쇼핑몰 도메인에는 다음과 같은 개념이 있을 수 있다.

- 회원
- 상품
- 주문
- 주문 항목
- 결제
- 배송
- 할인 정책

도메인 모델링은 이 개념들을 단순 데이터 구조가 아니라 책임을 가진 객체로 만드는 과정이다.

## 핵심

- 도메인 모델은 문제 영역의 중요한 개념을 표현한다.
- 객체는 데이터뿐 아니라 도메인 규칙도 함께 가져야 한다.
- Service가 모든 규칙을 처리하면 도메인 객체가 빈 데이터 덩어리가 된다.
- 객체 이름은 도메인에서 사용하는 용어와 맞추는 것이 좋다.
- 도메인 규칙이 어디에 있어야 자연스러운지 계속 질문해야 한다.

## 코드 비교

다음 코드는 주문 취소 규칙이 서비스에 들어간 예다.

```java
class Order {
    private OrderStatus status;

    OrderStatus getStatus() {
        return status;
    }

    void setStatus(OrderStatus status) {
        this.status = status;
    }
}

class OrderService {
    void cancel(Order order) {
        if (order.getStatus() == OrderStatus.SHIPPED) {
            throw new IllegalStateException("배송된 주문은 취소할 수 없습니다.");
        }

        order.setStatus(OrderStatus.CANCELED);
    }
}
```

주문 취소 가능 여부는 주문의 중요한 도메인 규칙이다. 이 규칙이 서비스에만 있으면 다른 코드에서 상태를 직접 바꿀 위험이 있다.

아래처럼 주문 객체가 취소 책임을 가지도록 만들 수 있다.

```java
class Order {
    private OrderStatus status;

    void cancel() {
        if (status == OrderStatus.SHIPPED) {
            throw new IllegalStateException("배송된 주문은 취소할 수 없습니다.");
        }

        status = OrderStatus.CANCELED;
    }
}

enum OrderStatus {
    ORDERED,
    SHIPPED,
    CANCELED
}
```

이제 주문 취소 규칙은 `Order` 내부에 모인다. 외부 코드는 `order.cancel()`을 호출할 뿐, 주문 상태를 직접 변경하지 않는다.

## 동작 흐름

1. 사용자가 주문 취소를 요청한다.
2. 애플리케이션 서비스는 취소할 `Order`를 찾는다.
3. 서비스는 `order.cancel()`을 호출한다.
4. `Order`는 자신의 상태를 확인하고 취소 가능 여부를 판단한다.
5. 취소 가능하면 상태를 `CANCELED`로 변경한다.

서비스는 흐름을 조율하고, 도메인 객체는 도메인 규칙을 판단한다.

## 주의할 점

도메인 모델링을 할 때 현실 세계를 너무 그대로 옮기려고 하면 복잡해질 수 있다.

코드의 객체는 현실의 모든 속성을 담는 것이 아니라, 현재 시스템이 해결해야 하는 문제에 필요한 책임만 가져야 한다.

또한 모든 로직을 도메인 객체에 넣어야 하는 것도 아니다. 외부 시스템 연동, 트랜잭션 관리, 저장소 접근 같은 애플리케이션 흐름은 Service나 Repository의 책임으로 두는 것이 자연스럽다.

## 정리

도메인 모델링은 문제 영역의 중요한 개념과 규칙을 객체로 표현하는 과정이다.

좋은 도메인 객체는 단순히 값을 담는 것이 아니라, 자신과 관련된 규칙을 스스로 처리한다. 이렇게 하면 비즈니스 로직이 한곳에 모이고, 서비스 코드는 전체 흐름을 조율하는 역할에 집중할 수 있다.
